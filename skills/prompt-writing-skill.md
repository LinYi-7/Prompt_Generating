# Prompt Writing Skill

## 适用范围

这是一套适用于“把原始内容整理成结构化输出”的提示词写法，尤其适合以下场景：

- 会议纪要
- 访谈整理
- 医疗记录
- 课堂笔记
- 培训分享
- 项目复盘
- 销售沟通
- 面试分析
- 用户研究

这套方法的核心不是“写得长”，而是让模型稳定地产出：

1. 结构清晰
2. 重点突出
3. 便于前端展示
4. 便于后端解析
5. 可直接复用

---

## 核心方法论

### 1. 先定义任务，不要先写格式

提示词第一部分先说明三件事：

- 模型扮演什么角色
- 输入里有什么
- 输出要解决什么问题

推荐写法：

```text
# 任务定义

你是一名专业的XXXX整理助手。你的任务是根据我提供的XXXX内容，生成一份XXXX。

输入内容可能包括：
1. ...
2. ...
3. ...

你需要以XXXX为主要依据，重点参考XXXX，输出一份标准、通用、便于阅读和后续处理的结果。
```

原则：

- 先说清“做什么”
- 再说清“依据什么做”
- 最后说清“输出要给谁用”

---

### 2. 基本要求里只放真正影响质量的规则

基本要求不是越多越好，而是只保留最影响输出稳定性的规则。

我常用的规则顺序：

```text
# 基本要求

1. 明确主要依据
2. 如果输入原文主要是什么语言，则输出遵循同样的语言
3. 不得编造未提及的信息
4. 用户手动标记或高亮内容优先处理
5. 输出遵循“先结果，后过程；先重点，后细节”
6. 内容按主题组织，不按流水账组织
7. 结构适合前端展示和后端解析
8. 信息不足时明确写“未提及”或“待确认”
```

经验：

- 语言跟随规则固定放在第 2 条
- “不得编造”必须出现
- “重点内容优先处理”必须出现
- “按主题组织”比“按发言顺序整理”更稳

---

### 3. 提取重点决定输出质量上限

模型最容易泛化失败的地方，不是格式，而是不知道“该抓什么”。

所以一定要单独写：

```text
# 提取重点

1. 主题、时间、人物
2. 核心目标或背景
3. 主要议题
4. 明确结论
5. 待办事项
6. 风险、分歧、未决事项
7. 核心数据与关键时间节点
8. 用户特别关注的笔记或标记内容
```

原则：

- 提取重点要覆盖“背景—讨论—结果—行动”完整链路
- 提取重点写得越准，模型越不容易跑偏
- 这部分比输出格式更影响成品质量

---

### 4. 输出结构先服务阅读，再兼顾解析

一份好的结构化输出，不是字段越多越好，而是要满足：

- 没参会的人能快速看懂
- 参会的人能快速回忆
- 前端能清晰展示
- 后端能稳定抽取

通用结构建议：

```text
## 会议信息
## 会议概览
## 主要议题
## 待办事项
## 风险与待确认事项
```

这套结构适合绝大多数会议和讨论类场景。

---

### 5. 主要议题不要写成流水账

这是最关键的一条。

低质量写法：

- 谁说了什么
- 然后谁又补充了什么
- 再然后谁提出了什么

高质量写法：

- 先按议题归纳
- 再在每个议题下提炼关键方面
- 再对每个关键方面做总结

推荐结构：

```text
## 主要议题

## 议题标题
- **关键方面1**：用一两句话总结
- **关键方面2**：用一两句话总结
- **关键方面3**：用一两句话总结
```

如果信息更复杂，再升级为：

```text
## 议题标题
- **关键方面1**
  1、具体信息点1
  2、具体信息点2
- **关键方面2**
  1、具体信息点1
  2、具体信息点2
```

使用原则：

- 普通纪要：优先“关键方面 + 总结句”
- 深度纪要：再展开到“具体信息点”
- 如果出现大量重复内容，必须归并，不要平铺

---

### 6. 核心数据与关键时间节点必须高亮

对会议纪要类内容，真正有价值的通常是：

- 时间
- 金额
- 比例
- 截止日期
- 版本号
- 目标值
- 结果值

所以要在提示词里显式要求：

```text
会议中提到的核心数据、关键比例、关键金额、关键版本号、关键时间节点、上线时间、截止日期等关键信息要加粗显示。
```

这样模型会自动把最值得扫读的信息突出出来。

---

### 7. 待办事项必须单独设计输出规则

如果不单独设计，模型很容易：

- 把讨论建议误写成待办
- 漏掉截止时间
- 漏掉执行人
- 写成一堆长句

推荐规则：

```text
对待办事项的抽取必须谨慎，只有在原文中存在明确动作、推进要求、后续执行语义、交付要求或时间线索时才提取为待办。
```

推荐输出：

```text
## 待办事项
任务内容（执行人）
截止时间：XXXX-XX-XX
```

如果执行人不明确：

```text
任务内容
截止时间：未提及
```

经验：

- 执行人不要单独拆字段时，就合并进任务标题
- 截止时间必须保留独立一行，方便后端解析
- 没有明确待办时，直接输出“暂无明确待办”

---

### 8. 用户手动记录内容要加权处理

如果输入里有用户手写笔记、标记时间点、重点标注，这些不是普通补充信息，而是“重要性信号”。

处理原则：

- 笔记 = 明确重点线索
- 标记 = 需要回看上下文的高价值信号

推荐规则：

```text
对于“笔记”，优先将其视为明确重点线索，并结合原文验证、归纳和扩展。
对于“标记”，必须围绕该时间点前后的相关原文进行分析，提炼对应的重要议题、结论、待办或风险。
```

这样模型才会真正“重视用户关注点”。

---

### 9. 引用来源要尽量贴近具体信息点

如果只是最后统一放一堆引用，后端和前端都不好用。

更好的方式是：

- 每个信息点后面都附引用来源
- 至少保留时间点
- 有条件的话保留转写 / 笔记 / 标记来源类型

推荐结构：

```text
1、具体信息点。
**引用来源**：
- transcript｜00:12:31｜原文摘要
- note｜00:12:40｜笔记原文
- marker｜00:12:45｜高光标记
```

原则：

- 引用来源不是装饰，而是可追溯性
- 前端跳转、后端解析、用户回看，都靠它

---

### 10. 一份好提示词要区分“已确定”和“待确认”

很多场景最容易出错的地方，就是把讨论写成结论。

所以要强制模型区分：

- 已明确达成的内容
- 讨论中的观点
- 待确认事项

推荐规则：

```text
必须区分：
- 已明确达成的内容
- 讨论中的观点
- 待确认事项
不要把讨论意见直接写成最终结论。
```

## 通用模板骨架

```text
# 任务定义
说明角色、输入、目标输出、使用场景

# 基本要求
1. 说明核心处理规则
2. 如果输入原文主要是什么语言，则输出遵循同样的语言
3. 说明禁止事项
4. 说明缺失信息如何处理
5. 说明是否保留不确定性/否定信息/冲突信息

# 提取重点
列出优先提取或优先判断的内容

# 输出要求
说明：
1. 只输出结果，不输出过程
2. 风格要求
3. 完整性要求
4. 给出输出格式示意
```

这是最通用、最稳的骨架。

## 会议纪要类提示词的推荐结构

```text
## 会议信息
| 项目 | 描述 |
|---|---|
| 会议主题 | ... |
| 会议时间 | ... |
| 参会成员 | ... |

## 会议概览
用一段话概括最重要内容，关键数据和关键时间节点加粗。

## 主要议题
## 议题标题
- **关键方面1**：总结句
- **关键方面2**：总结句
- **关键方面3**：总结句

## 待办事项
任务内容（执行人）
截止时间：XXXX-XX-XX

## 风险与待确认事项
- ...
```

## 适合沉淀成 skill 的一句话总结

这套 skill 的本质是：

**把原始内容整理类任务，统一拆成“任务定义—基本要求—提取重点—输出要求”四段，再用“先结果、后过程；先重点、后细节；按主题归纳而非按顺序复述”的方法，提高模型输出的稳定性、可读性和可解析性。**

---

## 使用建议

1. 简单任务用轻模板，不要过度约束。
2. 高价值任务一定要单独写“提取重点”。
3. 会议、访谈、课堂、医疗、销售这类场景，优先用“按主题归纳”。
4. 如果输出要给系统解析，就用固定字段标题。
5. 如果输出要给人快速看懂，就把核心数据和关键时间节点加粗。
6. 如果有用户标注内容，一定要提升权重处理。
